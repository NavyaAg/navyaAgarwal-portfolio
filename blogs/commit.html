How to write good commit messsages ?

Having reached till Mark 15, I noticed that I wasn’t efficient at passing good commit messages . They were either very vague or comprised several changes in one commit. As beginners we often tend to overlook these things but a good commit message is as important as the code. This blog tries to cover some conventional commit practices that we should follow.

Why good commit message matter?

  • It adds clarity to pull requests (PRs) raised- Most projects involve multiple people/teams working on different features. Each entity generally works on their local branch, which they rebase with the master at regular intervals. In such cases, having a broad idea of the code changes implemented by other collaborators via proper commit messages can save a lot of developer time spent in cross-communication.

  • It is key to effective debugging -Let’s say there’s a bug in the application which was not there before. To find out what caused the problem, reading the commit messages could be handy. The proper commit message can save a great deal of time finding the recent changes related to a bug.

  • It makes a code more readable and reusable-If we have a task to add some logic to some part of the code, previous good commit messages can help us find out where and how to add the code.

  • Makes tracking an implementation easier

  • Provide context to the code reviewer -A good commit message provides basic information to the reviewer about what the commit achieves, without them needing to dive into the code.

  • Help in maintaining a feature log-While writing release notes before shipping, developers can often quickly look at the history of commit messages to have a broad idea of the features added/removed/modified since the last release.



What is a good commit message ?

    The perfect commit message should have certain qualities:

  •  It should be understandable even by seeing only the header of the message

  •  Separate different changes to different commits-This is where I made major mistake. Because I forgot committing changes after every new functionality added , my messages often looked like-“js features added”

  • Should be reasonably small and atomic.It should be just enough, and not too detailed.

  •  It should be unambiguous.-The commit message should describe what changes our commit makes to the behavior of the code, not what changed in the code.

  •  Use the imperative, present tense - It is easier to read and scan quickly:

    Right: Add feature to alert admin for new user registration

    Wrong: Added feature ... (past tense)


Let’s now have a look at Conventional Commits Formats.

Formatting

type: subject

body (optional)

footer (optional)
  1. Type

    • feat- a new feature

    • fix- a bug fix

    • docs- changes in documentation

    • style- everything related to styling

    • refactor- code changes that neither fixes a bug or adds a feature

    • test- everything related to testing

    • chore- updating build tasks, package manager configs, etc

  2. Subject

    This contains a short description of the changes made. It shouldn't be greater than 50 characters.

  3. Body

    The body is used to explain what changes you made and why you made them. Not all commits are complex enough that they need a body, especially if you are working on a personal project alone, and as such writing a body is optional. A blank line between the body and the subject is required and each line should have no more than 72 characters.

There are several conventions used by different teams but this is by far most commonly used .
